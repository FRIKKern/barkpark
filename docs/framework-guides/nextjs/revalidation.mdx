Barkpark uses Next.js's tag-based cache with a deterministic tag shape,
so you almost never need to think about paths. The contract is:

| Tag shape                           | Invalidates                      |
| ----------------------------------- | -------------------------------- |
| `bp:ds:<dataset>:_all`              | Every request for a dataset      |
| `bp:ds:<dataset>:type:<type>`       | Every request for a document type|
| `bp:ds:<dataset>:doc:<id>`          | A single document                |

Every `barkparkFetch` call attaches the `_all` tag. `defineActions()` fans
out the `doc` and `type` tags after every mutation. You only need to reach
for `revalidateBarkpark()` when you are reacting to an event your own
action didn't trigger — typically a webhook.

## `revalidateBarkpark()` at a glance

```ts twoslash
import { revalidateBarkpark } from '@barkpark/nextjs/revalidate'

// Single id — treated as a document id.
revalidateBarkpark('p1')

// Rich payload.
revalidateBarkpark({ _id: 'p1', _type: 'post' })

// Bulk.
revalidateBarkpark({ ids: ['p1', 'p2'], types: ['post'] })
```

## From a webhook

Pair it with `createWebhookHandler` and forward the verified payload:

```ts twoslash
// @filename: app/api/barkpark/webhook/route.ts
import { createWebhookHandler } from '@barkpark/nextjs/webhook'
import { revalidateBarkpark } from '@barkpark/nextjs/revalidate'

export const { POST, GET } = createWebhookHandler({
  secret: process.env.BARKPARK_WEBHOOK_SECRET!,
  previousSecret: process.env.BARKPARK_WEBHOOK_SECRET_PREVIOUS,
  onMutation(payload) {
    revalidateBarkpark(payload as { _id?: string; _type?: string })
  },
})
```

Every Barkpark webhook delivery carries `_id` and `_type` fields; the
`revalidateBarkpark()` call translates those into the two canonical tag
calls (`doc:<id>` and `type:<type>`). Any subsequent RSC fetch for that
document or type will re-fetch at the edge.

## Time-based revalidation

When a webhook isn't available, pass `revalidate` to `barkparkFetch`:

```ts twoslash
// @filename: app/lib/barkpark.ts
import 'server-only'
import { createClient } from '@barkpark/core'
import { createBarkparkServer } from '@barkpark/nextjs/server'
const client = createClient({
  projectUrl: 'https://cms.example.com',
  dataset: 'production',
  apiVersion: '2026-04-12',
})
export const { barkparkFetch } = createBarkparkServer({ client, serverToken: 's' })

// @filename: app/posts/page.tsx
import { barkparkFetch } from '@/app/lib/barkpark'
import type { BarkparkDocument } from '@barkpark/core'

interface Post extends BarkparkDocument { title: string }

const posts = await barkparkFetch<Post[]>({
  type: 'post',
  revalidate: 3600,
})
```

Tag-based invalidation still wins when both are set — that's Next.js's
behaviour, not ours.

## Path-based revalidation (escape hatch)

`revalidatePath()` is not used by default because it invalidates
aggressively and is hard to compose with the RSC cache. If you really need
it (e.g. an infrequently-updated marketing route), opt in explicitly:

```ts twoslash
// Requires BARKPARK_ALLOW_ALL_REVALIDATE=1 in the environment.
import { revalidateBarkpark } from '@barkpark/nextjs/revalidate'

revalidateBarkpark({ path: '/', paths: ['/blog', '/about'] })
```

Without the env flag, the call throws. Keep the flag off in production
unless you have a specific reason to allow it.

## Checking your tags in dev

Run your RSC request, then inspect the `next.tags` attached to the outgoing
`fetch` in the server logs. A healthy read looks like:

```
fetch GET https://cms.example.com/v1/data/query/production/post
  tags: [ 'bp:ds:production:_all', 'bp:ds:production:type:post' ]
```

If the dataset tag is missing you are probably importing `barkparkFetch`
from the wrong module; if the `type:` tag is missing, pass it explicitly
via `opts.tags`.
